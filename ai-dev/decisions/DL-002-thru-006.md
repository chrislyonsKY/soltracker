# DL-002: Hybrid Data Strategy for Rover Traverses

**Date:** 2026-03-19
**Status:** Accepted
**Author:** Chris Lyons

## Context

Four rovers need traverse data. Two are active with live endpoints; two are retired with archival data only.

## Decision

**Hybrid approach**: live-fetch GeoJSON from NASA MMGIS for active rovers (Perseverance, Curiosity), pre-converted static GeoJSON committed to the repo for retired rovers (Spirit, Opportunity).

## Alternatives Considered

- **All live fetch** — Rejected. Spirit/Opportunity have no MMGIS endpoints; only PDS CSV archives exist.
- **All static** — Rejected. Would miss Perseverance/Curiosity traverse updates (new drives weekly).
- **Proxy server** — Rejected. Violates the "no backend" constraint for GitHub Pages.

## Consequences

- MER CSV → GeoJSON conversion is a one-time build-time ETL script.
- Active rover data stays current without manual updates.
- Static files add ~500KB to repo (acceptable).
- If MMGIS endpoints go down, active rover data shows last-cached version.

# DL-003: Calcite Design System for UI Framework

**Date:** 2026-03-19
**Status:** Accepted
**Author:** Chris Lyons

## Context

Need a UI framework for dashboard panels, controls, and responsive layout around the Mars SceneView.

## Decision

Use **Calcite Design System** (`@esri/calcite-components` v3.x) — Esri's web component library for consistent, accessible UI.

## Alternatives Considered

- **React + Tailwind** — Rejected. Adds React runtime overhead; ArcGIS v5.0 is web-component-native, so Calcite aligns better. Chris is familiar with React but Calcite provides tighter SDK integration.
- **Vanilla HTML/CSS** — Rejected. Would require reimplementing accessibility, responsive patterns, and dark mode from scratch.
- **Calcite + custom panels** — Considered but full Calcite is sufficient for v1.

## Consequences

- Calcite provides built-in dark/light mode toggle via `calcite-mode-dark` CSS class.
- Calcite Shell provides the responsive app layout out of the box.
- Calcite components are framework-agnostic web components — compatible with vanilla TS.
- Must import `@esri/calcite-components/dist/calcite/calcite.css` and call `defineCustomElements()`.

# DL-004: Vite Build Tooling

**Date:** 2026-03-19
**Status:** Accepted
**Author:** Chris Lyons

## Context

Need a build tool for TypeScript compilation, dev server, and static site output for GitHub Pages.

## Decision

Use **Vite 6.x** with TypeScript.

## Alternatives Considered

- **Webpack** — Rejected. Slower dev server, more complex config. The old `@arcgis/webpack-plugin` is unnecessary with v5.0.
- **No bundler (ES modules + import maps)** — Rejected. @arcgis/core has many internal modules; unbundled imports would be slow.

## Consequences

- Fast HMR during development.
- No special ArcGIS Vite plugin needed — @arcgis/core works with standard ES module resolution.
- Assets (WASM, workers) load from ArcGIS CDN by default.
- `vite build` outputs static files to `dist/` for GitHub Pages.

# DL-005: BYOK API Key Strategy with DEMO_KEY Fallback

**Date:** 2026-03-19
**Status:** Accepted
**Author:** Chris Lyons

## Context

NASA Photos API requires an API key. Esri Astro services are public. The app deploys as a static site — no server-side key management.

## Decision

- **NASA API key**: BYOK dialog in settings panel. User enters their free api.nasa.gov key, stored in localStorage. Falls back to `DEMO_KEY` (30 req/hour) when no key is provided.
- **Esri API key**: Optional. Design config to accept one if provided, but all current Mars services work without auth.

## Alternatives Considered

- **Build-time .env** — Rejected for GitHub Pages. The key would be in the built JS bundle, visible to anyone.
- **DEMO_KEY only** — Too restrictive (30/hour). Users hitting the photo gallery would exhaust it quickly.
- **Server proxy** — Rejected. No backend allowed.

## Consequences

- First-time users get limited but functional experience with DEMO_KEY.
- Power users provide their own free NASA key for higher limits.
- API key is stored only in user's browser (localStorage), never transmitted elsewhere.
- Settings dialog needs clear instructions: "Get a free key at api.nasa.gov".

# DL-006: NASA Mission Branding Colors for Rover Differentiation

**Date:** 2026-03-19
**Status:** Accepted
**Author:** Chris Lyons

## Context

All four rovers display simultaneously on the globe. They need distinct visual identities.

## Decision

Use colors derived from each mission's official branding materials:

| Rover | Primary | Secondary | Source |
|---|---|---|---|
| Perseverance | `#E03C31` (Mars Red) | `#C1440E` | Mars 2020 mission logo |
| Curiosity | `#3A7BD5` (Royal Blue) | `#4A90D9` | MSL mission patch |
| Opportunity | `#D4A843` (Gold) | `#C75B12` | MER patch gold tones |
| Spirit | `#2E8B57` (Sea Green) | `#1B3A6B` | Differentiation from Opportunity |

## Alternatives Considered

- **Single accent color with line style variation** — Rejected. Hard to distinguish on a complex 3D surface.
- **Arbitrary colors** — Rejected. Mission branding adds authenticity.
- **Stub for later** — Rejected. Colors affect the entire visual identity; better to decide early.

## Consequences

- All four rovers are visually distinct on any basemap.
- Color alone is not the sole differentiator (per WCAG) — rover labels and icons also distinguish them.
- Colors are defined in `src/config.ts` and referenced by all rendering code.
